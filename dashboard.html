Deep Research — What's Real, What's Hype, How to Start

Full brief in 05-deep-research.md
Claim-by-Claim Verdict: What's Real vs. What's Hype
The Claim Verdict Reality Check Source
"Ad creation: 30 min → 30 sec" Real* The 30 seconds is copy generation only — not strategy, brief writing, review, or upload. The full workflow from brief to live ad still takes human time. But the copy generation step genuinely went from 30 min to 30 sec. Anthropic Blog
"One person ran all of marketing" Overstated Austin ran performance marketing (paid search, paid social, SEO, ASO, email) solo. Brand marketing, product marketing, comms, influencer, customer marketing, and partner marketing had separate teams the entire time. He was "entire growth marketing" — not "entire marketing." Blog + X skeptics
"100+ ad variations per batch" Real Figma API supports programmatic frame swapping. The plugin takes copy from a sheet and generates permutations across aspect ratios. This is mechanically straightforward — it's automation, not AI magic. Anyone with Figma API access can replicate this. White Paper
"Sub-agent architecture is production-ready" Real Google Ads API (v18) and Meta Marketing API are fully available. Splitting headline vs. description agents by character constraint is a proven pattern. MCP servers for both platforms exist. This works today. White Paper
"Self-improving A/B memory system" Early Described as a "rudimentary memory system" by Anthropic themselves. It's a JSONL log injected into context — not a sophisticated learning loop. Useful, but "self-improving" oversells it. It's more like "context that remembers." White Paper
"5x digital marketing productivity YoY" Unverifiable Self-reported by Anthropic about their own product. No methodology disclosed — what's the baseline? What counts as "productivity"? This is a marketing claim inside a marketing case study. Treat as directionally true, not precise. Anthropic Blog
"Non-coder built a Figma plugin in 45 min" Plausible Claude Code can scaffold Figma plugins — the Figma API is well-documented. But 45 min assumes no debugging, no edge cases, no iteration. Realistic first version: probably 2-4 hours including setup. Still impressive for a non-coder. TechFlow
"This scales to an agency with 150 clients" Not Yet Austin's architecture is single-tenant, single-brand. Scaling requires: credential isolation per client, context isolation (no cross-client bleed), a job queue with observability, rate-limit-aware retry logic. A broken workflow at scale = 150 broken outcomes. Level analysis
"Brand voice consistency via AI" Hard LLMs are good for direct-response copy. Weak for idiosyncratic brand voices. Austin's guardrails were written by PMM and copywriters first — the AI runs on top of human-approved guidelines, not instead of them. His own quote: "sometimes it doesn't get it right on the first try." Anthropic Blog
Bottom line: The automation of repetitive, rule-based, API-connected tasks (ad copy generation, creative variations, reporting pulls) is real and replicable today. The narrative that "one person replaced an entire marketing department with AI" is marketing spin on a genuine but narrower achievement. The hardest problems — brand voice at scale, multi-client isolation, and quality assurance — are unsolved and require human judgment.

How to Start This Week — 3 Steps
1
Run the POC on 1 real client. Have their media buyer review 50 outputs. Track approve / edit / reject. That dataset is more valuable than any code you write this week.
2
Add a compliance-check agent as a third pass after headline + description agents. Catches character limit violations, banned words, and missing disclaimers before human review.
3
Wire approved outputs into a JSONL hypothesis log. On next run, inject the last N approved/rejected entries as a "client memory summary." This is the beginning of the self-improving loop.
Memory System Design
JSONLUse now · weeks 1–4
One file per client. Append each experiment: client ID, copy, approval status, edits, notes. Inject last N entries as context summary per run.
SQLiteMigrate at 30 days or 1,000 records
When you need cross-client queries: "all rejected headlines in finance vertical." No server required.
Vector StorePhase 3 only
Semantic retrieval of historically similar approved copy. Real value — but adds embedding pipeline complexity. Don't build until JSONL proves value.

Replication Playbook — What to Steal and How

Week 1-2: Copy the Core
1. Build a /rsa slash command with three Agent Skills: brand voice, product accuracy, platform constraints
2. Create two sub-agents: Headline Agent (30 chars) + Description Agent (90 chars)
3. Wire CSV input → sub-agents → CSV output
4. Run on 1 real client — have their media buyer review 50 outputs and track approve/edit/reject rates
Week 3-4: Add Memory
1. Create a JSONL hypothesis log per client (client ID, copy, approval status, edits, notes)
2. Inject last N approved/rejected entries as context summary per run
3. Add a compliance-check agent as third pass: catches character limit violations, banned words, missing disclaimers
4. Connect Meta Ads MCP server for real-time campaign analytics without platform switching
Month 2+: Scale & Compound
1. Migrate JSONL → SQLite at 1,000 records for cross-client queries ("all rejected headlines in finance vertical")
2. Build Figma plugin for creative variations (100+ per batch)
3. Add multi-channel orchestration: agent per channel with coordinator above
4. Phase 3: Vector store for semantic retrieval of historically similar approved copy
Model selection tip from Anthropic: Use Haiku for volume generation (ad variations, bulk copy) and Sonnet for brand judgment (tone review, compliance checks). This mirrors Austin's sub-agent split — cheap fast agents for production, smarter agents for quality gates.
